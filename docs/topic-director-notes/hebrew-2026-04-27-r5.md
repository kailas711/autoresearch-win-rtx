## 2. On-thesis assessment of Phase 1 investigation

**Are the 3 RCs accurately diagnosed?**

Yes. The hex evidence in Section B is dispositive. For ЧђЧћЧЁ (entry 469), Ч™Ч“Чў (entry 2353), ЧўЧ©Ч” (entry 4977), and Ч‘Ч•Чђ (entry 743), the source binary contains no `vb.` label anywhere in the entry вЂ” confirmed by full-body scan, not just the first 200 bytes. The two prior hypotheses from r4 are eliminated by this evidence. The 298 entries that ARE correctly classified all have an explicit `vb.` in body at entry decode time; the ~1,200+ missing verbs simply omit this label by HALOT editorial convention.

**Is RC1's binyan-stem inference approach safe? Could it over-classify?**

The approach is safe. The inference rule fires only when explicit binyan stem markers (`qal:`, `niphal:`, `piel:`, etc.) are present in the entry body вЂ” specifically, the `\n\t\t{stem}:` lowercase-with-two-tabs pattern already used by `HalotExtensionHooks::extract_binyanim`. Non-verb entries (nouns, adjectives, prepositions) do not contain these markers. The investigation shows 0 over-extraction in the current state (non-verbs have no binyanim). Adding an inference rule that fires on the binyan-marker signal should not create false positives: anything that has a `qal:` section IS a verb by HALOT's own structure. The Verifier-4 over-extraction check (non-verb entries with binyanim must equal 0) enforces this bidirectionally.

**Is the RC1+RC2 coupling correct?**

Yes. Section C of the investigation confirms that both RCs affect the same code path: `split_halot_line` в†’ `extract_grammar` в†’ `normalize_grammar_label`. RC2 (Roman numeral stub misfiring) causes incorrect headword extraction for 1,294 homograph entries, which means their body text starts at the wrong position вЂ” which in turn affects whether RC1's binyan-stem scan sees the correct body content. Fix RC2 first (correct the headword/body split for `"I "` format), then apply RC1's inference rule on the corrected body. Doing them in separate commits risks the RC1 rule operating on mis-split bodies for the homograph entries. Co-dispatch is correct.

**Is RC3 truly independent of RC1+RC2?**

Yes. RC3 (false-positive `n.` matches) operates in `extract_grammar`'s pattern matching, not in the headword-split or the grammar-inference path. Fixing RC1+RC2 changes what is in `fields.body` for some entries, but RC3's defect вЂ” that the `n.` pattern matches etymological abbreviations like `"MHb."` вЂ” is orthogonal to whether the body was correctly split or whether a binyan marker was found. Section C of the investigation confirms the `n.` match occurs independently via `pattern_list` substring search. RC3 can be in a separate commit without interaction risk.

**In fellwork-api `crates/fw-binaries/src/decoder/`:**

Step 2 вЂ” RC1 (do second, larger): Add a binyan-stem inference rule. If any stem label is present, set `grammar = "vb."` and `grammar_normalized = "verb"` before proceeding to `classify_from_grammar`. This allows the binyan extractor to receive these entries correctly.

The binyan-stem scan itself already exists in `HalotExtensionHooks::extract_binyanim`. The RC1 fix lifts a version of that scan upstream into the grammar classification step, not as a replacement but as a fallback when explicit `vb.` is absent.

Commit message format: `fix(halot): RC1+RC2 binyan-stem inference + Roman-numeral stub for HALOT implicit verbs (investigation: fix/halot-grammar-investigation@859aebf)`

### Phase 2b вЂ” RC3 (separate commit)

**In fellwork-api:**

Tighten the `n.` pattern match in `extract_grammar` (or in `halot.toml`'s grammar pattern definitions). The current substring match fires on `n.` anywhere in the body, including inside etymological abbreviations (`"MHb."`, `"OSArb."`, `"EgArm."`, `"JArm."`). Requirement: `n.` must be word-boundary-anchored вЂ” preceded by whitespace, start-of-body, or a delimiter (semicolon or comma), AND must not appear as a trailing part of a longer abbreviation token. The simplest safe implementation: require the character before `n.` to be a space, tab, newline, or start-of-string, and require the character after `.` to be a space, tab, newline, or end-of-string. Do not match `n.` within multi-character abbreviations.

Commit message format: `fix(halot): RC3 tighten n. pattern boundary to suppress etymological false positives (investigation: fix/halot-grammar-investigation@859aebf)`
