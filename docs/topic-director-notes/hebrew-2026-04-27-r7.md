#### C. Token-id format spec

**Hebrew format:** `<book>.<chapter>.<verse>.<token_index>`

- `book` — canonical 3–4 char abbreviation matching the key prefix in `genesis_full_hebrew.json` (e.g., `Gen`, `Exod`, `Lev`).
- `chapter` — 1-indexed integer, no zero-padding.
- `verse` — 1-indexed integer, no zero-padding.
- `token_index` — 0-indexed integer within the verse, counting canonical Hebrew word-tokens as defined by OSHB tokenization.

Example: `Gen.1.1.0` = first Hebrew word of Genesis 1:1.

**Greek format (specified now for schema parity; not implemented in r8):** Identical structure with NT book abbreviations: `Matt`, `Mark`, `Luke`, `John`, `Acts`, `Rom`, `1Cor`, etc. The Greek track token-id space is orthogonal to Hebrew; no collision possible.

**Edge cases the Architect must specify handling for:**

- **Ketiv / Qere:** The written form (ketiv) gets the canonical `token_id` and `surface`. The qere (marginal reading) is stored in `annotation_flags: ["ketiv"]` on the token plus a companion field `qere_surface` (nullable). The qere does NOT get its own `token_id`.
- **Proclitic prepositions / conjunctions:** OSHB tokenization may split `בְּרֵאשִׁית` as a single token or two. Architect must specify: follow OSHB tokenization as the canonical split decision. Each OSHB token unit = one `token_id`. Document the OSHB tokenization boundary decision explicitly.
- **Maqaf-joined compounds:** `WORD1־WORD2` — treat as separate tokens following OSHB token boundaries (maqaf is a connector, not a space, but OSHB typically preserves word boundaries). If OSHB treats them as one token, they share one `token_id`.
- **Elision / textual lacunae (DSS variant readings):** Out of scope for token_id generation; flagged via `annotation_flags` and `manuscript_witnesses`.

**Stability contract:** given the same OSHB version and ETCBC morphology version, the same input verse must always produce the same `token_index` ordering. `token_id` is stable across re-extractions when the source version is pinned. Version pinning is enforced by recording the OSHB/ETCBC commit SHA in a corpus manifest file alongside the JSONL outputs.

#### D. Fingerprint algorithm spec

**Algorithm:** BLAKE2b with 16-byte (128-bit) digest (`hashlib.blake2b(digest_size=16)`). Output stored as hex string (32 lowercase hex chars).

**Input construction:**
1. Collect the fingerprint-included fields (see below) in sorted field-name order.
2. For each field: `<field_name>=<value>` where value is the canonical string representation.
3. Join pairs with `\x00` (null byte) separator.
4. Encode the joined string as UTF-8.
5. Compute BLAKE2b-128 digest.

**Fields IN scope for fingerprint** (these determine token structural identity):

| field | representation |
|---|---|
| `token_id` | verbatim string |
| `surface` | verbatim Unicode string |
| `lemma` | verbatim consonantal Hebrew string |
| `morphology_code` | verbatim ETCBC tag string; empty string if null |
| `strong_number` | zero-padded 4-digit form `H0000`; empty string if null |
| `gk_number` | verbatim; empty string if null |

**Fields OUT of scope for fingerprint** (these can change without changing token structural identity):

- `annotation_flags` — curator-supplied, not structural.
- `halot_key`, `bdb_key`, `km_number` — lexicon join targets; can be updated as lexicons are re-extracted without invalidating the token.
- `gloss_en`, `catss_alignment` — derived annotations.
- `phrase_role`, `phrase_depth` — Macula annotations that may be updated independently.
- `passage_id` — passage grain might change without the token changing.
- `source_provenance` — administrative.

**Stability requirement:** same OSHB version + same ETCBC morphology version + same STEPBible TSV version (for `strong_number`) ⇒ same fingerprint. Version SHAs are recorded in the corpus manifest (not in the per-token record). When any versioned source updates, a fingerprint diff report shows which tokens changed; those tokens' curated annotations require re-review.

**Python reference implementation (spec; not production code):**

```python
import hashlib

FINGERPRINT_FIELDS = [
    "token_id", "surface", "lemma", "morphology_code",
    "strong_number", "gk_number"
]

def compute_fingerprint(token: dict) -> str:
    parts = []
    for field in sorted(FINGERPRINT_FIELDS):
        val = token.get(field) or ""
        parts.append(f"{field}={val}")
    payload = "\x00".join(parts).encode("utf-8")
    return hashlib.blake2b(payload, digest_size=16).hexdigest()
```

The Architect must verify: (a) no two tokens in Gen.1.1 should produce the same fingerprint (since `token_id` is in scope); (b) two tokens with identical `surface`, `lemma`, `morphology_code` but different `token_id` values will produce different fingerprints by construction.

#### E. Passage grain decision

**Three options with tradeoffs:**

| option | primary advantage | primary disadvantage |
|---|---|---|
| per-verse | 1:1 with existing `genesis_full_*.json` key format; no boundary decisions | too fine for passage-level metadata (genre annotation on every verse independently = huge curation burden; nearly identical across adjacent verses) |
| per-pericope | natural unit for authorial context and literary devices; matches how commentaries discuss passages; Macula Hebrew has discourse-unit boundaries | requires pericope boundary decisions; non-trivial automation |
| per-chapter | lowest curation burden; chapter boundaries already known | chapters are a medieval scholarly division, not an authorial unit; conflates distinct narrative episodes |

**Director preference: per-pericope as primary unit.** Pericopes are the natural unit for all passage-level metadata fields: genre, subgenre, literary devices, author tradition, era. Assigning these per-verse is redundant (adjacent verses share the same genre); assigning them per-chapter is too coarse (Gen.1 contains both the creation narrative and the day-structure). Macula Hebrew provides discourse-boundary annotations that can seed pericope definitions mechanically, with manual review for ambiguous cases.

**Architect must choose one option and provide written justification** against all three. If the Architect chooses per-verse or per-chapter, the justification must explicitly address why the per-pericope advantage is outweighed.

**If per-pericope is chosen (recommended):**
- Define `passage_id` format: `<book>.<pericope_index>` where `pericope_index` is a 1-indexed integer within the book (e.g., `Gen.1` = creation week, `Gen.2` = second creation account, etc.). Architect proposes specific index boundaries for at least Gen.1–11 as validation.
- Pericope source of truth: Macula Hebrew discourse annotations as the mechanical default; canonical heading-based override for ambiguous cases.
- Per-token record carries `passage_id` (pericope FK) AND the original `token_id` (which encodes verse address). The verse address is always recoverable from `token_id` even when the passage_id is pericope-grained.
- Per-passage record carries `chapter_start`, `verse_start`, `chapter_end`, `verse_end` for range lookup from verse refs.

**Cross-canon link grain:** same decision as passage grain for the source side. Target refs use the token_id verse notation (e.g., `John.1.1` for the verse-level link target) since NT pericope boundaries will be decided in the Greek track round.

#### F. Sourcing pipeline

For each schema field, name the source(s) and conflict-resolution rule when sources disagree.

**Authoritative source hierarchy:**

1. OSHB (Open Scriptures Hebrew Bible) — canonical Hebrew text + ETCBC morphology. Source of truth for `surface`, `lemma`, `morphology_code`, `strong_number` (via embedded Strong's if present; otherwise STEPBible TSV join).
2. STEPBible TSV — `github.com/STEPBible/STEPBible-Data` Strong's↔GK Hebrew mapping. Source of truth for `gk_number` and cross-lexicon joins. Vendor as data file; do NOT code-generate.
3. Macula Hebrew — `github.com/Clear-Bible/macula-hebrew` — phrase-node and discourse annotations. Source of `phrase_role`, `phrase_depth`, pericope boundary suggestions.
4. Sefaria API — alternate vocalized Hebrew text. Secondary backup for `surface` when OSHB tokenization is ambiguous. Cleanup required (HTML entities, editorial annotations per `docs/domain-knowledge-cache.md`).
5. Accordance `.agloss` (`MT-E words.agloss`) — English glosses. Source of `gloss_en`. Offset-driven parser per domain-knowledge-cache.
6. Accordance `.ainter` (`MT-LXX Interlinear.ainter`) — CATSS parallel alignment. Source of `catss_alignment`. Tar archive; verse-ref table at offset `0x5A400C` per domain-knowledge-cache.
7. Lexicon fixtures (`lex_bdb.json`, `lex_halot.json`, `lex_km.json`) — source of lexicon FK joins. Join via `strong_number` → `lex_bdb.strongs`, `lex_halot.strongs`, `lex_km.strongs`. Secondary join via `lemma` → `headword_consonantal` for entries where Strong's is null.

**Conflict resolution rules:**

- **`surface` conflict (OSHB vs Sefaria):** OSHB wins. Sefaria is backup only; flag disagreements in `annotation_flags: ["sefaria-surface-divergence"]`.
- **`morphology_code` conflict:** Only OSHB/ETCBC is used; no alternate morphology source.
- **`strong_number` conflict (OSHB embedded vs STEPBible TSV):** STEPBible TSV wins for Strong's↔GK mapping; OSHB embedded Strong's wins if STEPBible has no entry for a token.
- **`phrase_role` / `phrase_depth` from Macula:** single source, no conflict. Mark as null if Macula coverage absent for a verse.
- **Multi-witness flag:** `passage_metadata.manuscript_witnesses` lists which traditions cover the passage. For passages with known LXX / DSS divergence, add to `passage_metadata.annotation_flags`. Divergent alternate readings are NOT represented as additional token rows in v1 — that is a later curation pass. v1 reflects MT only.

**Source versions must be pinned and recorded in a corpus manifest** (`data/corpus/hebrew/manifest.json`) with:
- OSHB commit SHA
- Macula Hebrew release tag
- STEPBible TSV commit SHA
- lex_bdb / lex_halot / lex_km fixture generation dates

#### H. Alternatives considered

**H.1 — Flat sentence-pair table (rejected)**
A flat table of (Hebrew verse, English verse) pairs is what `prepare_translate.py` currently uses. It is sufficient for a baseline chrF score but eliminates all structural signal: no per-token identity, no morphology, no phrase role, no lexicon augmentation, no passage genre. The model trained on flat pairs cannot distinguish Qal from Niphal, narrative from poetry, or Pauline from Johannine register. Rejected per `feedback_corpus_dimensions.md` design intent.

**H.2 — Per-token without passage table (rejected)**
Storing token-level data without a passage metadata table loses the authorial/historical dimensions. The model would have per-token morphology but no genre signal, no era signal, no register signal. These are exactly the features that differentiate gold-quality translation from mainstream NMT output (per `feedback_corpus_dimensions.md`). Rejected.

**H.3 — Document database (MongoDB / SQLite) vs flat JSONL**
Document databases enable richer query but add a server dependency, schema migration overhead, and remove grep-ability. JSONL is portable, grep-able, diff-able, can be loaded incrementally with a single `for line in file` loop, and can be re-generated from scratch. For a research corpus where the team iterates on the schema and wants to inspect records directly, JSONL wins on tooling simplicity. If query performance becomes a bottleneck at corpus scale, JSONL can be loaded into SQLite via `pandas.read_json()` or `duckdb` without changing the canonical format.

**H.4 — Single combined table vs separate token + passage tables**
A single table would embed all passage metadata on every token row. For a corpus of ~470K tokens × ~30 passage metadata fields, this is ~14M redundant field instances. Every passage-level annotation change requires re-writing all tokens in the passage. The two-table design normalizes correctly: one passage row is updated, all tokens automatically reflect the change via the FK. Rejected per `feedback_corpus_dimensions.md` explicit guidance.

**H.5 — Strong's as primary lexicon join key vs consonantal lemma**
Strong's numbers are coarser than consonantal lemmas (one Strong's number can cover multiple homographs). Consonantal lemma is more precise but requires string matching against `headword_consonantal` in the fixtures, which has known edge cases (Roman numeral prefixes, Aramaic entries). The spec adopts Strong's as the primary join key (most reliable for automation) with consonantal lemma as fallback for entries where Strong's is null in the fixture. The Architect should confirm this hierarchy against the actual fixture join rates.

#### I. Migration / backfill path

**Existing fixtures are unchanged.** `data/fixtures/genesis_full_hebrew.json`, `lex_*.json`, `gold/catalog.json` remain on disk in their current format. `prepare_translate.py` and `score_translate.py` continue to read them without modification. Plan #6 corpus is an additive new artifact at `data/corpus/hebrew/`.

**Key namespace alignment:** `genesis_full_hebrew.json` uses `Gen.1.1` as a verse key. Plan #6 `token_id` format is `Gen.1.1.<index>`. The prefix `Gen.1.1` is identical. This means:
- `score_translate.py`'s verse-ref parsing (`ref_start.split(".")` on the gold catalog) is compatible with Plan #6 token_id namespace.
- Future revisions to `prepare_translate.py` that consume Plan #6 corpus can locate all tokens for a verse by prefix-filtering `token_id` on `<verse_ref>.`.

**Future `prepare_translate.py` revision path:** v2 of prepare_translate will read `data/corpus/hebrew/v1.jsonl` instead of flat verse strings, join per-token morphology and phrase annotations, and produce enriched BPE sequences carrying structural special tokens (per the phase-1 spec: `<phrase>`, `<clause>`, `<construct>`, suffix tags). This revision is out of scope for Plan #6 and r8; it is the natural r10+ work after the corpus exists and passes Verifier acceptance.

**Corpus versioning:** the `v1` in the filename is the corpus assembly version, not the source-data version. Corpus versions increment when the schema or assembly logic changes. Source versions are tracked in the manifest file (§ F). This makes diffs between corpus versions auditable.

#### J. Out-of-scope (explicit)

The following are explicitly NOT deliverables for Plan #6 / r8:

1. **Training pipeline rewrite** — `prepare_translate.py` is not modified in r8. The corpus is assembled; consumption by the training pipeline is a separate later round.
2. **Greek track corpus** — schema parity is specified in this document (§ C, § A identifiers) but no Greek corpus assembly is initiated until Hebrew first model is trained.
3. **Curation tooling** — sense disambiguation, manual binyan annotation, and gold-alignment passes are separate tooling rounds. Plan #6 corpus carries `annotation_flags` as an empty list; curation attaches later.
4. **Cross-canon link generation** — `cross_canon_links` field exists in the per-passage schema but is initially empty or null for all rows. Populating it is an annotation pass; separate round.
5. **Accordance `.atext` parser changes** — the `.atext` files (`MT-ETCBC-A`, `GNT28-T`) require new OakTree-reader code per the domain-knowledge-cache. This is a separate decoder work item. Plan #6 uses OSHB (public, GitHub-hosted) for morphology, not the Accordance `.atext` binary.
6. **KM binyan audit** — KM is acceptable at iter-4 baseline for Plan #6. Binyan coverage audit is deferred.
7. **Disputed passage authorship curation** — the `author_tradition` field is specified and schema-present but populated with mechanical defaults (e.g., `["Mosaic"]` for Pentateuchal passages) in v1. Detailed source-criticism curation is a separate annotation pass.
