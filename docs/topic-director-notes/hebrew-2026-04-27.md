## 1. On-thesis assessment of pending pile

**Plan #6 — corpus assembly (token_id + fingerprint + passage_metadata + lexicon augmentation)**
HIGH impact, but PREMATURE. This is the direct path to training signal — without it the model has no lexicon features and no per-token identity. However, building Plan #6 on top of a BDB with 1.1% binyan coverage embeds a structural defect at the foundation. Binyanim are the primary morphological discriminator for Hebrew verb semantics (Qal vs Piel vs Hiphil change meaning, not just voice). A corpus assembled now would need to be re-assembled after #7b is fixed. Plan #6 is HIGH-priority to reach eventually, but BLOCKED on #7b.

**Plan #7b — BDB binyan coverage (1.1%)**
HIGH priority, HIGH urgency. 1.1% is not a gap — it is an absence. If ~50%+ of Hebrew verbs lack binyan extraction, the per-token morphology dimension in Plan #6 is hollow for verbs. This is a signal-degradation finding, not a fixable nuisance. The defect is bounded: single extractor, single field class, known source format. It should be investigated and fixed before Plan #6 runs. Priority: first dispatch this session.

**Plan #7c — HALOT multi-stem (natan/shaphat missing)**
MEDIUM priority. HALOT binyanim are already at 95.6% on verbs (iter-4). The multi-stem gap is real but narrower than the BDB absence. It affects specific high-frequency verbs (נתן, שפט are both extremely common in Genesis). This should be the second Mode 3 defect fix after #7b closes, ideally in the same session.

**Plan #7a — BDAG GK (Greek Mounce) 0%**
LOW for Hebrew track. Greek track is currently Pending status. This is on-thesis for the eventual Greek track but has zero bearing on Hebrew-track corpus assembly or training. File for the Greek track's first session.

**Acceptance criteria (runnable):**
```python
import json, collections
data = json.load(open('data/fixtures/lex_bdb.json', encoding='utf-8'))
verbs = [e for e in data if e.get('pos') in ('verb', 'v', 'verb (qal)', 'vb') or 'verb' in str(e.get('pos','')).lower()]
with_binyan = [e for e in verbs if e.get('binyanim') and len(e['binyanim']) > 0]
print(f"Verb entries: {len(verbs)}")
print(f"With binyan: {len(with_binyan)} ({100*len(with_binyan)/max(len(verbs),1):.1f}%)")
# Named samples — verify these specific high-frequency verbs:
named = ['אמר', 'הלך', 'נתן', 'שמע', 'ראה', 'ידע', 'בוא', 'שוב', 'עשה', 'דבר', 'קרא', 'שאל']
for lemma in named:
    matches = [e for e in data if e.get('lemma') == lemma]
    print(f"  {lemma}: {len(matches)} entries, binyanim: {[e.get('binyanim') for e in matches]}")
```

## 7. Continuity check

**Anti-patterns to watch:**

1. **Target revision (failure #1):** If Verifier reports "BDB only has 500 verb entries, so 1.1% is expected given the source," check against the known 8,846 total entries. BDB is a comprehensive Hebrew lexicon — expect 1,000+ verb entries. Do not accept revised targets.

2. **Aggregate stats hiding sample failures (failure #5):** The binyan audit MUST run named samples. "Coverage 40%" means nothing if נתן and אמר (the two most frequent Hebrew verbs in Genesis) both score 0.

3. **Deferred items as PASS (failure #3):** If the Builder produces a fix that works for Qal verbs but defers Niphal/Piel/etc., that is a PARTIAL, not a PASS. Binyanim are multi-valued per lemma.

4. **HALOT iter-4 pattern as false analogy:** HALOT binyan fix (0.1% → 95.6%) worked because the source had `\n\t\t{stem}:` markers. BDB may have a completely different encoding. The Builder must investigate first; do not assume the same fix applies.
